
ch.5 雲端安全
```
雲平台風險和責任共擔
--未經授權和非法數據PII被披露的最後法律責任是擁有數據的雲客戶
--雲提供商可能在財務上承擔全部或部份、政府不會對提供商求償
--法規可能要求雲客戶承擔政府、數據主體的特定損失、但可能最終從雲廠商得到相應補償
--
--
--雲客戶關心數據
-->雲數據中心產生環境是客戶生命線、違規失職和可用性缺失是對雲客戶最大影響
--雲提供商關心
-->數據中心的安全和營運、生存和維持營利的能力方式
-->客戶想強制推行政策、獲取日誌數據、對數據中心的性能和安全進行審計

--私有雲 分布式計算和BYOD功能 ，私有雲面臨風險
-->人員威脅  無意惡意威脅，但還在組織控制下滿意程度處理好方法
-->自然災害  精準掌握事件準備情況，備份頻率、類型、地點
-->外部攻擊  未授權訪問、竊聽、拒絕服務
-->監管不合規 可對內部全面控制、準確了解存在監管風險並有信心合規
-->惡意軟體  取決於感染來源

--社區雲  共享和分散資源、共同/單獨/集中/跨社區
-->通過共享所有權或的彈性 因網路所有權和營運分在客戶之間，大量節點受損但不影響其他節點
-->>很實現一致配置管理，分散式的所有權也帶策略和管麵的分散式決策。

-->成本分擔  分散基礎架構開銷成本，訪問和控制也是如此
-->不需要堆性能和艦測進行集中化管理 消除集中化管理負擔，但失去標準質量和安全監測帶來的可靠性

--公有雲   對雲服務依賴性，到期移出費用更多
-->雲服務供應商綁定 portablity"可移植性" 轉移難易程度
>>>確保有利於移植的合同條款  提早考慮推出策略，試用打折、提前轉移懲罰、執行困難
>>>避免轉有格式  原始數據再另朵雲可用格式恢復，否則不要簽約
>>>確保轉移沒有物理限制  離開十足夠寬帶轉移整個數據及合，新服務商可以這個量級數據導入
>>>檢查監管限制 若需要獨特具有限制性(FERPA,PCI,HIPAA)能使用雲供應商很有限

-- 雲服務供應商鎖定  可能被收購或退出服務
-->提供商生命週期  從業多久、領導者、中間公司可能無預警退出
-->核心競爭力  滿足所需、員工和資源是否滿足，雲服務公司是核心業務還是附屬業務
-->司法管轄權的適用性  在哪個國家、特許地點和運營地點、數據中心在哪、儲存和備份能力"數據會跨邊界邊境、能滿足所有適用法規。
-->供應鏈的依賴性  依靠其他任何實體獲得關鍵能力、存在必要供應商銷售、公用事業設施萬一缺少就無法提供服務。
-->立法環境 特定法規可能影響使用該提供商、難預測潛在影響

--多租戶環境  組織應保持警惕此方式，因規模無法擴大不會盈利
-->利益衝突 不應同是競爭對有關聯，應避免這類情況。Brewer-Nash模型(利益規避)
-->特權提升 常識或去未授權，非法管理拳用戶可能控制那些處理其他客戶的數據的設備
＊＊  
-->信息洩露  多客戶通過湘婷基礎設施處理和儲存數據，可能會被其他客戶接收讀取。比如說處理數據時間過長。
-->法律活動  刑事案件作為證據的部分，可能包含其他租戶數據一起被查封在同一件硬件設備。

Brewer-Nash  利用職責分離、最小權限方指利益衝突可能性、制定支持和實施模型策略解決這個問題
BLP 每組數據分級和自己許可級別限制訪問數據級和

>>>混合雲

--IaaS
-->人員威脅  惡意或疏忽的內部人員、因他們可物理訪問
-->外部威脅  惡意軟件、黑客、拒絕、分布式拒絕、中間人
-->缺乏特定技能  大量需自行管理，需要具備足夠雲環境培訓人員，否則業務運營面臨風險


--PaaS
-->互操作性問題  由雲復霧提供的管理跟更新，環境新調整可能可用性低
-->持續後門 適合做為新應用的測試平台DevOps，但產品發布會有留下開發留下後門
-->>> 為高效即可查找要解決的特定功能
-->虛擬化 PaaS
-->資源共享 同設備運行，信息洩漏和側信道攻擊。

--SaaS
-->專有格式  供應商有為一格式收集、儲存、顯示數據，導致雲服務供應商綁定降低可值性
-->虛擬化 SaaS虛擬化風險會加劇，更多資源供想和多租戶的狀況
-->web應用安全 SaaS依賴瀏覽器訪問。具有 (API)接口。 web應用程行潛在弱點

虛擬化
```

--攻擊虛擬機管理程序
-->>> 可以攻擊虛擬機管理程序本身、底層操作系統。
--客戶機逃逸  Host Escape
-->>> 配置不當允許用戶突破現志離開自己虛擬機，可能性不大但是必須考慮的風險
--信息洩露
-->>> 失效或故障引起風險，同一主機。特定時間其他機器正在進行操作
--數據封查
-->>> 被執法機構或原告律師沒收檢查

雲攻擊面 部署模式威脅

--私有雲
-->惡意軟件  透過網路上傳下載到內部網路的惡意軟體導致數據丟失設備失去控制權、操作中斷
-->內部威脅  內部員工或被授予訪問權限個人的惡意意外活動造成的結果
-->外部攻擊  因經濟利益、黑客、政治目標或感到不滿而攻擊網路 DDoS 數據洩漏和法律後果
-->中間人攻擊  採取簡單竊聽獲取數據或更先進攻擊去修改數據流量能力講虛假或有害引入
-->>> 私有雲遠程訪問增加這類威脅暴露

-->社會工程  易受社會工程威脅、增加潛在攻擊向量、
-->>> 因用戶需要管理幫助重製密碼大量使用郵件通進分配訪問令牌和加密密鑰

-->盜竊丟失設備  用戶設備BYOD的丟失盜竊可能導致私有雲為授權訪問和利用
-->監管違規  私有雲增加阻止無法維護合規性風險、信息傳機會及效率的提高也增加違反相關法規的可能性
-->自然災害  物理位置和氣候決定各種猜害的類型和頻率，攻擊數據中心本身或公用電力網路設施
-->>> 快速複製、定期備份、分不是、元成處理、儲存數據帶來的安全性

--社區雲
-->失去策略控制  所有權分布在社區雲中無法實現集中策略頒布和實施
-->失去物理控制  意味成本低也是控制力度下降的物理安全性的降低
-->審計訪問權限不足  分布式環境進行稽核可能是不切實際或不可能

--公有雲
-->流氓管理員  比基本訪問句更大權限的內部人員可能以惡意或不負責任方式行事造成更多損害且不受客戶控管
-->特權提升  內部威脅的延伸、授權用戶的惡意操作提高權限級別、為了違反也要執行自己任務能力
-->>> 用戶面對至少兩個治理方式(組織、供應商)、導致請求修改或授予額外訪問權限延遲導致用戶嘗試繞過政策
-->合同失敗  較差合同可能導致供應商綁定的不利條款、缺法必要服務和其他風險

--混合雲
>>> 具其他形式部署全部威脅、存在由於失去統一和集中化控制所帶來的風險
```

對策
```
>>>惡意軟件  提供有關惡意軟體進入雲還問方法和如何防護的具體培訓
-->>> 不間斷持續間惡流量和基線備至、可檢測受感染的異常活動和性能下降、定期更新補丁、自動更新檢查
＊＊12
>>>內部威脅  積極背調、定期培訓、強制休假、崗位輪調、財務運營雙人操作模式
-->>>   職責分離、最小權限的可信賴工作流程、提供數據混淆這播處理、DLP
>>>外部攻擊者 可信賴的安全機線、縝密配置變更管理、強健方問控制措施、可外包CASB的第三方
-->>> 可用於微些評估和識別和一些預測能力、比被動式微些處理更即時響應

>>>中間人攻擊 加密傳輸中的數據、身份驗證
>>>社會工程攻擊  辦公室的安全
>>>設備盜竊丟失 加密的儲存資料、嚴格物理訪問控制、限制或關閉USB功能、庫存控制和監測

>>>違反規定 僱用知識豐富、受訓冰具備相關技能員工、在規劃和管理聽從法律顧問建議、實施DRM根據需要加密混淆和遮蔽技術
>>>自然災害  系統和服務的冗余性、使用同個或不同供應商或離線方式做DR
>>>失去策略控制  採用強有力合約條款、供應商至少遵守用戶一樣控制和有效完整的安全計畫、第三方審計
>>>失去物理控制  條目列出所有保護措施
>>>審計訪問權限不足  依靠可信第三方、雲客戶在合同上堅持體現保護、盡可能把安全的財務責任給雲提供商並有額外陳罰損害賠償
>>>流氓管理員  所有特權帳戶和人員的附加物理、邏輯和管理措施的表列、管理所有管理活動完整和安全紀錄、特權做財務的持續監視

>>>特權提升 使用身份驗證工具和技術全面的訪問控制、安排培訓、技能收斂人對日誌數據進行分析審查、使用SIEM解決方案
>>>合同問題  防止供應商綁定鎖定、考慮完全異地備份到雲客戶或信任第三方保護和保管、避免重構
>>>法律查封  法律訴訟導致組織突然或意外丟失或披露、修訂BIA的可能性、考慮雲端數據進行加密

特定BIA關注點
-->需重新檢視現有BIA、雲帶來新風險和機會進行部分評估、遷移後失去連接產生可能的危害
>>>新的依賴關係 數據和操作依賴外部(供應商、供給商、公共設施)
-->BIA應考慮提供商無法滿足服務要求的可能性、相關實體存在類似問題
>>>監管失敗  雲數據的分發效率和易用性增加潛在違規可能性、不滿足法規要求
--> 雲提供商可能無法或不願遵守造成無法滿足，PH, ePHI, GLBA,HIPAA,FERPA,SOX
--> BIA需要分析這種情況產生的影響

>>>數據洩漏 內部人員遠程訪問、需再評估為授權洩漏的衝擊和影響
--> 信息洩露帶來潛在不利影響導致合同違約
>>>供應綁定鎖定 遷移雲端任何操作
-->BIA 首次考慮遷移作為成本效益分析

客戶與提供商分擔BC/DR責任

--備份數據系統邏輯位置
-->私有架構 雲服務作為備份 BC/DR使用雲提供商作為備份
>>>定期上傳帶寬的費用、備份頻率、使用備份模式、數據中心和系統的安全性和網路服務的費用
>>>客戶可以決定切換時間點、觸發緊急、正常操作會停止，將備份端當作正式工作網路
>>>容災切換可能作為遠程網路(IaaS,PaaS,SaaS)、或是下載到另一個站點應急操作直到恢復正常的方式和時間

-->雲端營運 雲供應商作為備份  提供商可提供備份解決方案作為服務內容、通常是透明的
>>>一個其他地理位址作為災難的數據中心作為備份、完全由供應商負責備份物理位置、備份配置及評估和宣布災難事件
>>>切換時客戶有極少量參與但更屬於另外不是常態、 作為服務特性提供的一部分，BC/DR通常不會有額外費用

-->雲端營運 第三方雲備份 換到另朵雲、分散風險增加冗余度
>>> 主動將供應商綁定鎖定的可能性、這可能需複雜協商BC/DR的模式、角色和責任必須明確徹底規劃
>>> 供應商跟雲客戶都會參與緊急事件的評估和宣布、容災切換需要共同參與
>>>由於合作和溝通困難會遇到障礙、雲客戶必須與主要和備份雲服務協商列表(私有架構、雲做備份)的所有條款
>>>因費用不會綑綁計費、容災時換和應及操做都需要額外費用、利用為滿足SLA要求主要供應商抵銷部分增加費用

--災難通告 BC/DR事先決定誰將會被授權作出決定及明確的通告過程
-->應接受詳細應急的操作作為培訓、包括組織具體的BC/DR計畫的廣泛深刻的理解、
>>>應由高層管理授權人員、具宣佈緊急情況並啟動切換流程的充分能力
>>>應具預警系統評估即將發生的災難、事件發生前進行容災切換保持連續性
>>>雲供應商和客戶通知形式達到一致、可建立預備性溝通初步計劃安排
>>>合同應規定收到通知後必須完成時間、或是由客戶完全控制需要明確說明
>>>組織有特定實體負責、做出決定人員應充分瞭解其內在風險和影響並在合同中說明

-->恢復測試  備份是滿足法規要求、履行應盡關注due care義務、嘗試使用這些備份可用性
>>>BC/DR 應注意確認和分配具體相關責任
